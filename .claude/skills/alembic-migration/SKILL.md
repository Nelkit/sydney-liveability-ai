# Alembic Migration Skill

## When To Use
- You need to add or modify a field in `backend/db/models.py`.
- You need to generate a new Alembic revision for a schema change.
- You need to apply or verify database migrations for PostgreSQL.

## Context
- ORM models live in `backend/db/models.py`.
- Migrations are managed by Alembic in `backend/alembic/`.
- `make db-upgrade` applies all pending migrations.
- Database connection comes from `DATABASE_URL` in `backend/.env`.
- Never apply raw SQL directly in Supabase to add columns; always use Alembic so the schema stays in sync for the team.

## Critical Patterns
- Add the ORM field first, then generate the migration.
- For new columns on existing tables, default to `nullable=True` unless the table is being recreated.
- Review autogenerated `upgrade()` and `downgrade()` before applying.
- Commit the model change and generated migration together.
- Use descriptive migration names that explain the change.

## Full Workflow
1. Update the SQLAlchemy model in `backend/db/models.py`.
2. Generate a migration with `alembic revision --autogenerate`.
3. Inspect the revision file in `backend/alembic/versions/`.
4. Apply the migration with `make db-upgrade`.
5. Verify the database is at head and the column exists.
6. Tell teammates to run `make db-upgrade` after pulling the branch.

## Decision Rules
- If the table already has data: make the new column nullable.
- If the change is a rename or type change: inspect autogenerated SQL carefully; autogenerate may need manual edits.
- If the migration would drop data: stop and ask for confirmation first.
- If the change affects ingestion scripts or agents: update those files after the schema is migrated.

## Quality Gates
- ORM change exists before migration generation.
- Generated migration file is reviewed before apply.
- Migration description states the actual change.
- `make db-upgrade` succeeds without pending migrations afterward.
- No raw SQL schema edits are used as a shortcut.

## Template
```python
# backend/db/models.py
class OsmScore(Base):
    __tablename__ = "osm_scores"
    suburb: Mapped[str] = mapped_column(String, primary_key=True)
    osm_score: Mapped[float | None] = mapped_column(Float, nullable=True)
    cafe: Mapped[int | None] = mapped_column(Integer, nullable=True)
    supermarket: Mapped[int | None] = mapped_column(Integer, nullable=True)
```

```bash
cd backend
alembic revision --autogenerate -m "add supermarket to osm_scores"
make db-upgrade
```

## Commands
```bash
cd backend
alembic revision --autogenerate -m "add <field> to <table>"
make db-upgrade
alembic current
alembic history
alembic downgrade -1
```

## Integration Notes
- Keep migrations small and one-purpose.
- Update any scripts or agents that write to the changed table.
- If the schema change is required for ingestion, migrate before re-running ingestion scripts.
